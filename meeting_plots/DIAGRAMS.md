## HAL Architecture Diagram

**Purpose:** Show the dispatch path difference between the old (upstream) CUDA HAL and the new (cuda_new) HAL. Explains *why* dispatch overhead dropped 4-19x.

**Mermaid source:** `hal_architecture.mmd`

**To render:**
```bash
# Option 1: Mermaid CLI
npx @mermaid-js/mermaid-cli -i hal_architecture.mmd -o hal_architecture_rendered.png -b transparent

# Option 2: Mermaid Live Editor
# Paste contents of hal_architecture.mmd into https://mermaid.live
# Export as PNG/SVG
```

**Diagram spec (for manual recreation in Figma/PowerPoint/LaTeX):**

See `hal_architecture_spec.txt` for full text description.

Summary of what to draw:

```
OLD CUDA HAL (~1ms/dispatch):
+---------------+    +-------------------+    +---------------+    +------------+    +----------------+
| VM Bytecode   | -> | Deferred Command  | -> | Queue         | -> | Stream     | -> | cuLaunchKernel |
|               |    | Buffer            |    | Emulation     |    | Sync       |    |                |
+---------------+    +-------------------+    +---------------+    +------------+    +----------------+
                            |                        |
                     [CUDA graph capture]    [Multi-queue scheduling]


NEW CUDA HAL (~0.06ms/dispatch):
+---------------+                                                    +----------------+
| VM Bytecode   | -------------------------------------------------> | cuLaunchKernel |
|               |                                                    |                |
+---------------+                          +------------------+      +----------------+
                                           | Event            |
                                           | Synchronization  |
                                           +------------------+
```

**Key labels:**
- Old path: 4 layers between VM and GPU. Supports graph capture + multi-queue.
- New path: Direct launch. Single stream. Event-based sync.
- Trade-off: No CUDA graph capture, no multi-queue. Kept: NCCL, async memory pools.

**Color scheme:** Old = warm (orange/red border). New = cool (green border).

### Representative Matmul

All plots use **2048x2048 f32** as the representative matmul size (typical hidden dimension at GPT-2 scale). This size is:
- Compute-bound enough to show codegen quality differences
- Large enough that dispatch overhead is small but measurable
- Our strongest result: cuda_tile 1.5x faster than LLVMGPU

### Style Notes

- Minimal text on plots — annotations for poster go in presentation layer
- Consistent colors: blue=LLVMGPU, orange=cuda_tile, green=torch.compile
- Red-Yellow-Green heatmap for sweep (green=fast, red=slow)
- Log scale on latency axes where range > 10x
